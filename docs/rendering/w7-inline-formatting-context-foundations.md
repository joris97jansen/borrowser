## Purpose

W7 makes inline formatting context data explicit generated-box metadata. The
goal is not complete CSS inline layout correctness yet; it is to define a
stable contract between box generation and line building so later intrinsic
sizing, overflow, text, and fragmentation work can extend the inline model
deliberately.

The current pipeline shape is:

```text
StyledNode
  -> DisplayBoxGeneration
  -> BoxTree with parent/child, containing-block, block-context, and inline-context links
  -> LayoutBox geometry projection
  -> inline token collection
  -> line box construction
  -> block/inline geometry refinement
```

Inline formatting context assignment belongs to layout-owned box generation.
Line building consumes the generated metadata; it does not infer inline
participation from DOM tag names or raw `BoxKind` alone.

## Current Model

Each generated `BoxNode` records:

- `inline_formatting_context: Option<InlineFormattingContextId>`
- `establishes_inline_formatting_context: bool`
- `inline_formatting_participation: InlineFormattingParticipation`

`InlineFormattingContextId` is a frame-local wrapper around the generated
`BoxId` that establishes the inline formatting context. It is intentionally
distinct from DOM node ID, parent `BoxId`, `ContainingBlockId`, and
`FormattingContextId`.

The metadata is projected into `LayoutBox` so inline token collection and later
geometry algorithms can use the same contract without rebuilding `BoxTree`.

## Establishment Rules

The following generated boxes establish an inline formatting context in the
current supported subset:

- document-element, block, and list-item boxes whose direct generated children
  form an inline-only child set
- anonymous block boxes generated by W4 for contiguous inline-level runs inside
  mixed block/inline containers
- inline-block boxes whose current supported child set directly forms an
  inline formatting context

The following generated boxes do not establish an inline formatting context:

- document root boxes
- ordinary inline boxes
- text-run boxes
- replaced inline boxes
- marker boxes
- inline-block boxes with mixed inline/block children until their internal
  mixed-flow model is explicitly supported
- block-level boxes whose direct generated children are all block-level
- block-level boxes whose mixed children are normalized through anonymous block
  boxes

This keeps the model explicit: a block container owns line building only when
its generated child set directly forms an inline formatting context. When W4
anonymous blocks are generated, the anonymous boxes own the line-building scope
for their wrapped inline runs.

## Inline Participation

`InlineFormattingParticipation` records the supported inline-flow role of each
generated box:

- `InlineContainer` participates as an inline box whose inline descendants are
  collected into the current line-building scope.
- `TextRun` participates as text content in the current line-building scope.
- `AtomicInline` participates as one atomic inline-level item, such as
  inline-block and currently supported replaced inline boxes.
- `None` marks boxes that do not participate directly in an inline formatting
  context.

This metadata is separate from `BlockFormattingParticipation`. A box can be
block-level in its parent block formatting context while also establishing an
inline formatting context for its own inline children.

## Line Building Contract

`collect_inline_tokens_for_block_layout(...)` only emits tokens when the layout
box establishes an inline formatting context. Token collection then uses
`InlineFormattingParticipation` for each descendant:

- `InlineContainer` recurses into inline descendants.
- `TextRun` emits text tokens from the sourced text node.
- `AtomicInline` emits a single atomic inline token and does not descend into
  children for the parent line.
- `None` terminates the current inline run boundary for whitespace handling.

This prevents line-building code from relying on incidental DOM shape or the
temporary fact that text-run geometry currently projects through
`BoxKind::Block`.

## LayoutBox Projection

The geometry projection preserves the W7 metadata:

- `LayoutBox::inline_formatting_context()`
- `LayoutBox::establishes_inline_formatting_context()`
- `LayoutBox::inline_formatting_participation()`

Future sizing, overflow, and text algorithms can consume inline-context
relationships from `LayoutBox` without retaining the original generated
`BoxTree`.

## Debug Surface

`BoxTree::to_debug_snapshot()` and `LayoutPhaseOutput::to_debug_snapshot()`
include:

- `ifc=<box-id|none>`
- `establishes-ifc=<yes|no>`
- `inline-participation=<none|inline-container|text-run|atomic-inline>`

These fields are part of the deterministic regression surface for W7.

## Deferred Work

W7 intentionally does not implement:

- full CSS inline formatting context correctness
- line box tree or fragment tree separation
- bidi reordering
- vertical writing modes
- full whitespace, wrapping, and text shaping behavior
- intrinsic inline sizing
- inline overflow and clipping behavior
- ruby, text decoration propagation, or generated-content layout
- retained inline formatting context identity across layout generations
- flex, grid, table, or fragmentation contexts

Those features must extend inline context establishment, participation,
line-building contracts, debug snapshots, and tests deliberately.

## Regression Surface

The layout crate tests cover:

- inline-only block containers establishing inline formatting contexts
- inline boxes and text runs participating in the nearest inline formatting
  context
- inline-block boxes participating atomically in the parent context while
  establishing an internal inline formatting context for descendants
- mixed-child inline-blocks remaining atomic in the parent inline formatting
  context without claiming unsupported internal direct IFC ownership
- anonymous block boxes establishing inline formatting contexts for wrapped
  inline runs
- projection of inline-context metadata into `LayoutBox`
- deterministic debug snapshots exposing inline-context metadata
